# BlockQuant

> **Quantize HuggingFace LLMs to EXL3 on a local NVIDIA GPU or in the cloud — and publish them to HuggingFace — from a CLI, a Discord slash command, or a FastAPI endpoint.**

[![license: MIT](https://img.shields.io/badge/license-MIT-blue.svg)](LICENSE)
[![python: 3.10+](https://img.shields.io/badge/python-3.10%2B-blue.svg)](#prerequisites)
[![format: EXL3](https://img.shields.io/badge/format-EXL3-c63010.svg)](https://github.com/turboderp-org/exllamav3)
[![runpod](https://img.shields.io/badge/cloud-RunPod-7c3aed.svg)](https://runpod.io)
[![docker image](https://img.shields.io/badge/ghcr-blockquant-2a2620.svg?logo=docker)](https://github.com/Honkware/blockquant/pkgs/container/blockquant)

BlockQuant adds three layers on top of [ExLlamaV3](https://github.com/turboderp-org/exllamav3)'s
`convert.py`: a **provider abstraction** that runs the same job locally or on
a RunPod H100/A100, a **live browser dashboard** that visualises the 48-layer
× 256-expert grid during quantization, and **post-upload tooling** (model
cards, HF collections, config fixups) that follows the bartowski / turboderp
/ ArtusDev publishing convention.

## Showcase

Models quantized with this tool, published under the
[`blockblockblock` HF account](https://huggingface.co/blockblockblock):

- **Qwen3.6 · 35B-A3B · Claude 4.7 Opus Reasoning Distilled** — 5 EXL3 variants (3.0 / 4.0 / 4.5 / 5.0 / 6.0 bpw) → [collection](https://huggingface.co/collections/blockblockblock/qwen36-35b-a3b-claude-47-opus-reasoning-distilled-exl3-69ece89b196fa4ae78d37550)
- **Qwen3.5 · 9B · Claude 4.6 Opus Reasoning Distilled** — 4/6/8 bpw EXL3
- Plus a backlog of smaller research builds; see [all `exl3` models on the account](https://huggingface.co/blockblockblock/models?search=exl3) for the full list

Each repo carries a templated card (real on-disk size, sibling-bpw table, VRAM
hints, recipe details) generated by `backend/scripts/publish_quant.py`.

## Prerequisites

| What | Version | Notes |
|---|---|---|
| Python | 3.10+ | tested on 3.10/3.11/3.12 |
| NVIDIA GPU | any with recent CUDA | only required for the **local** provider |
| Node.js | 20+ | only required for the **Discord bot** front-end |
| Redis | any recent | only required if you use the FastAPI / Celery / Discord stack |
| HuggingFace token | write-scope | needed to upload quants |
| RunPod API key | — | only required for the **cloud** provider |

You can use any single front-end (CLI, Discord, or HTTP API) — the others are
optional. The simplest path is the **RunPod CLI**: it doesn't need Node, Redis,
Celery, or a local GPU.

## Quickstart — RunPod CLI (recommended)

The minimal entry point. Requires only a Python virtualenv, a RunPod
account, and an HF token. No Discord bot, no local GPU, no Redis.

```bash
git clone https://github.com/Honkware/blockquant.git
cd blockquant

# Vendored upstream — kept out of this repo for licensing clarity
git clone https://github.com/turboderp-org/exllamav3.git

cd backend && python -m venv venv
./venv/bin/pip install -r requirements.txt
cd ..

cp .env.example .env
# Set: HF_TOKEN=...   RUNPOD_API_KEY=...
$EDITOR .env

# Quant a 9B model at 4.5 bpw on a community-tier H100 NVL
./backend/venv/bin/python backend/scripts/run_runpod_job.py \
  --model Qwen/Qwen2.5-7B-Instruct \
  --variants 4.5 \
  --gpu "NVIDIA H100 NVL" \
  --gpu-fallback "NVIDIA H100 PCIe,NVIDIA A100-SXM4-80GB" \
  --hf-org ""    # blank = upload to your personal HF account
```

The pod boots, bootstraps the stack, downloads the model, runs ExLlamaV3's
`convert.py`, uploads each `--variants` value to its own HF repo, **self-terminates**,
and exits. Out-of-stock GPU types fall back through the `--gpu-fallback` list
automatically. Typical end-to-end time + cost:

| Model size | GPU tier | Wall clock | Cost (community / secure) |
|---|---|---|---|
| 7-9B | H100 NVL | ~30-45 min | ~$2 / $5 |
| 30-35B (MoE) | H100 SXM HBM3 | ~3-4 h | ~$10 / $30 |

### Speedup presets

```bash
./backend/venv/bin/python backend/scripts/run_runpod_job.py \
  --model ... --variants 4.5 \
  --profile fast              # cal_rows=128 + community tier
  # or --profile balanced     # cal_rows=250 (default)
  # or --profile quality      # cal_rows=512 + secure tier
```

`--cal-rows` and `--cloud SECURE|COMMUNITY` override the preset's individual knobs.
Add `--tune` to print the resolved config + cost estimate without launching.

### Pre-baked Docker image (skip bootstrap)

The default path bootstraps a base RunPod image (~5 min for apt + pip + JIT
compile). To skip that, point at the pre-baked image published from this repo:

```bash
./backend/venv/bin/python backend/scripts/run_runpod_job.py \
  --image ghcr.io/honkware/blockquant:latest \
  --model ... --variants 4.5
```

The image bakes the validated `(torch, CUDA, ninja, ExLlamaV3, formatron, ...)`
combination + a `flash_attn` import shim. Builds are pushed to GHCR
automatically on every `v*.*.*` git tag — see
[`.github/workflows/build-runpod-image.yml`](.github/workflows/build-runpod-image.yml).
Set `BLOCKQUANT_RUNPOD_IMAGE` in `.env` to make it the default for every run.

## Quickstart — local GPU

Same setup as above, but pass `provider=local` (Discord) or use the local
pipeline directly. Requires a CUDA-capable NVIDIA GPU and the ExLlamaV3 clone
already on disk.

```bash
./backend/venv/bin/python -m blockquant.pipeline run \
  --model meta-llama/Llama-3.1-8B-Instruct \
  --bpw 4.5 \
  --provider local
```

The local provider runs the full pipeline (download → quantize → upload)
in-process. Use it when you want to keep the job out of the cloud entirely.

## Quickstart — Discord bot (advanced)

The Discord front-end adds a queue, EXP scoring, history, and a slash-command
UX on top of the same pipeline. Use this path when running quants as a
service for a server or community; the CLI is the simpler choice for one-off
jobs.

```bash
npm install                                 # in the repo root

./backend/venv/bin/redis-server &
./backend/venv/bin/python -m celery -A scheduler.tasks worker --loglevel=info -P solo &
./backend/venv/bin/python -m uvicorn api.main:app --host 127.0.0.1 --port 8000 &
node src/index.js
```

Required `.env` keys: `BOT_TOKEN`, `CLIENT_ID`, `GUILD_ID`, `HF_TOKEN`,
`EXLLAMAV3_DIR`. In the server: `/quant url:meta-llama/Llama-3.1-8B-Instruct
bpw:4.5 provider:local`.

Slash commands — full set:

| Command | Description | Access |
|---|---|---|
| `/quant` | Submit a quant job (local or RunPod) | All |
| `/queue`, `/health`, `/score`, `/leaderboard`, `/history` | Status / scoring | All |
| `/give`, `/pause`, `/resume`, `/cache`, `/diag` | Admin / ops | Admin |

## Live dashboard

`backend/scripts/log_dashboard.py` is a single-file FastAPI app. Tails the
quant log, runs a regex-based **LogParser** that emits typed events, derives
stats (tensor rate, sec/layer, runtime, spend, RunPod balance) anchored on log
timestamps + file mtime so they survive a refresh, and streams state to the
browser via Server-Sent Events.

```bash
./backend/venv/bin/python backend/scripts/log_dashboard.py \
  --port 8088 \
  --logs-dir backend/logs \
  --log-pattern "runpod-*.log"
# → open http://localhost:8088
```

The strip at the top shows every active job (pip-style stage indicators,
percent, ETA); clicking a tile swaps the detail panel below. The detail panel's
centerpiece is a live render of the model's expert matrix — cells settle from
cream (pending) → vermillion (measuring) → sage (quantized) in real time.

## What's in the box

| Layer | Purpose |
|---|---|
| **Discord bot** (`src/`) | `/quant` slash command, EXP/queue/history |
| **FastAPI + Celery** (`backend/src/api/`, `backend/src/blockquant/`) | Job orchestration, stage runners, progress reporting |
| **Providers** (`backend/src/blockquant/providers/`) | `local` (any NVIDIA GPU) and `runpod` (H100/A100 cloud). Lambda + Modal are shelved — see [`experimental/README.md`](experimental/README.md) |
| **CLI** (`backend/scripts/run_runpod_job.py`) | One-shot RunPod job without the Discord stack |
| **Dashboard** (`backend/scripts/log_dashboard.py`) | Browser UI + expert-matrix visualisation |
| **Publishing** (`backend/scripts/publish_quant.py`, `fix_repo_card_and_config.py`) | Templated model cards + HF collection management |
| **Pre-baked image** (`docker/`) | Pinned-stack RunPod image, built by GHA on tag |

## Pipeline stages

1. **Download** — HF snapshot pulled to workspace
2. **Convert** — GGUF only (EXL3 reads HF directly; experimental, EXL3 is the supported path)
3. **Quantize** — ExLlamaV3 `convert.py`, streaming progress
4. **Verify** — load-test each variant
5. **Quality** *(optional, EXL3 only)* — KL-divergence + perplexity vs FP16 baseline via `exllamav3/eval/model_diff.py`
6. **Report** — model-card README with measured quality
7. **Upload** — one repo per bpw: `{org}/{model}-exl3-{bpw}bpw`

Cloud providers run stages 1-5 remotely; the dashboard reads the tailed log.
The local provider runs in-process. Both respect `--keep-pod` for debugging
or rescue uploads.

## Contributing

**New provider?** Subclass
`backend/src/blockquant/providers/base.Provider`. The ABC defines three
required methods (`launch`, `terminate`, `run`) plus seven optional hooks
with safe defaults — implement only what your backend needs. Use
`runpod_provider.py` as a reference (SSH + SFTP + retry-on-transient-network-error
throughout, plus a self-terminate path so pods don't burn credit if the local
client is paused or dies).

All new providers must ship a matching test file under
`backend/tests/providers/` modelled on `test_runpod_provider.py`. Run the
suite with `pytest backend/tests/providers -q` (currently 33 passing, no GPU
required).

Before bumping the Dockerfile or the in-pod stack, scan
[`docker/Dockerfile.runpod`](docker/Dockerfile.runpod)'s comments — every
non-obvious pin is annotated with the failure mode that motivated it.

## Security

The FastAPI server has **no authentication** by design — it binds to
`127.0.0.1` and is reached only by the local Discord bot or CLI. **Do not
expose port 8000 publicly** without a reverse proxy + auth in front; anyone
who reaches it can submit jobs against your RunPod credit and your HF
account.

The RunPod provider uses paramiko's `AutoAddPolicy` for SSH host keys.
This is intentional — each pod is ephemeral with a fresh key, so strict
checking would amount to throwing the integrity check away anyway. Pod
identity is verified through the RunPod control plane (the `pod_id` we
asked for) and traffic is encrypted over SSH.

Report security issues privately to the maintainer rather than via a public
issue.
